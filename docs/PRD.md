# SmartFill PRD (Product Requirements Document)

> 版本：v0.1 MVP  
> 更新时间：2026-02-05  
> 产品定位：AI驱动的PDF自动填写工具

---

## 1. 背景与问题

### 1.1 核心问题
在日常生活中，人们经常需要填写各种PDF表单：租房合同、保险文件、申请表格等。传统方式要么依赖电脑端的PDF编辑软件（Adobe Acrobat），要么需要打印-手写-扫描。这些方式在移动场景下效率低下，特别是当用户只有手机、时间紧迫时，体验极差。

### 1.2 用户洞察
目标用户画像：
- 年龄：20-30岁
- 职业：学生、初入职场的打工人
- 场景：需要填写租房合同、保险表单、学校申请等
- 频率：几个月一次（低频但确定性需求）
- 痛点：
  - 手边没有电脑时无法方便地填写PDF
  - 重复填写相似信息（姓名、身份证、地址等）效率低
  - 现有工具要么太重（需要装软件），要么不够智能（手动点击每个字段）

### 1.3 现有解决方案的不足
- Adobe Acrobat：需要安装，移动端体验差，价格贵
- 在线PDF编辑器：需要手动点击每个字段填写，没有智能化
- ChatGPT等AI助手：只能识别文件内容，无法直接输出填好的PDF
- 无人解决的空白：没有产品做到"上传PDF → AI理解并自动填写 → 输出可用文件"的完整闭环

## 2. 产品目标

### 2.1 短期目标（MVP阶段）
- 验证"AI自动填写PDF"这个核心价值是否成立
- 完成技术可行性验证，确保AI能准确识别和填写常见表单
- 获得第一批种子用户反馈（技术圈、独立开发者社区）

### 2.2 成功指标
- 技术指标：AI填写准确率 > 80%（常见表单）
- 用户指标：有10个真实用户完整使用一次流程
- 开发指标：3周内完成可用MVP

### 2.3 非目标（明确不做什么）
- ❌ 不追求商业化或大规模用户增长
- ❌ 不做复杂的用户系统和数据存储
- ❌ 不做企业级功能（批量处理、团队协作等）
- ❌ 不追求所有场景覆盖，只做常见可编辑PDF

## 3. 核心功能（MVP v1.0）

### 3.1 功能列表

#### ✅ 必须有（P0）

F1. PDF文件上传
- 用户可以通过Web界面上传PDF文件
- 文件大小限制：< 10MB
- 支持格式：可编辑PDF（带表单字段的PDF）
- 移动端和桌面端均支持拖拽/点击上传

F2. 对话式信息输入
- 用户通过文字对话框输入需要填写的信息
- 输入方式：自然语言（如"我叫张三，身份证号123456..."）
- 系统不限制输入格式，由AI自动理解和提取

F3. AI智能填写
- 后端调用大模型（GPT-4或Claude）：
  1. 识别PDF中的表单字段（字段名、类型、位置）
  2. 将用户输入的信息与字段进行语义匹配
  3. 自动填充对应字段
- 填写逻辑：尽可能填满所有可识别的字段

F4. 填写结果下载
- 生成填好的PDF文件
- 用户可直接下载
- 文件名自动生成：`原文件名_filled.pdf`

F5. 响应式UI
- 支持桌面浏览器（Chrome、Edge、Safari）
- 支持移动端浏览器访问
- 界面简洁，核心流程不超过3步

#### ⏸️ 暂不实现（留待后续版本）
- ~~PDF预览界面~~
- ~~对话式修改/重新填写~~
- ~~点击编辑功能~~
- ~~扫描版PDF识别（OCR）~~
- ~~语音输入~~
- ~~用户信息记忆（Memory）~~
- ~~批量处理~~

### 3.2 用户流程

```
1. 用户打开网页
   ↓
2. 上传一个PDF文件（可编辑PDF）
   ↓
3. 系统提示："请告诉我需要填写的信息"
   ↓
4. 用户在对话框输入：
   "姓名：张三
    身份证：123456199001011234
    地址：北京市朝阳区xxx
    电话：13800138000"
   ↓
5. 点击"开始填写"按钮
   ↓
6. 系统显示loading状态（预计10-30秒）
   ↓
7. 生成填好的PDF，自动下载
   ↓
8. 用户打开PDF确认
   - 如果满意：完成
   - 如果不满意：重新上传文件，重新输入信息
```

### 3.3 错误处理

上传阶段：
- 文件格式错误 → 提示"仅支持PDF格式"
- 文件过大 → 提示"文件大小不能超过10MB"
- 不是可编辑PDF → 提示"暂不支持扫描版PDF，请上传可编辑的PDF文件"

填写阶段：
- AI识别失败 → 提示"无法识别表单字段，请确认这是一个标准表单"
- AI填写失败 → 提示"填写失败，请检查输入信息或稍后重试"
- 超时（>60秒） → 提示"处理超时，请稍后重试"

## 4. 技术实现方案

### 4.1 技术栈（推荐）

前端：
- React 18 + TypeScript
- Tailwind CSS（快速响应式布局）
- Axios（文件上传和API调用）
- 部署：Vercel

后端：
- Python + FastAPI
- PyPDF2 或 pypdf（PDF字段读写）
- OpenAI API（GPT-4）或 Anthropic API（Claude）
- 部署：Railway / Render

存储：
- 不做持久化存储
- 文件处理完即删除
- 无需数据库

### 4.2 核心技术挑战

挑战1：PDF字段识别
- 解决方案：使用PyPDF2读取PDF的AcroForm字段
- 备选方案：如果无法读取字段，提示用户这不是可编辑PDF

挑战2：信息匹配准确性
- 解决方案：通过Prompt Engineering优化AI匹配逻辑
- 示例Prompt：
  ```
  你是一个PDF表单填写助手。
  PDF中的字段：["姓名", "身份证号", "联系电话", "现住址"]
  用户提供的信息：{用户输入}
  
  请将用户信息匹配到对应字段，输出JSON格式：
  {"姓名": "张三", "身份证号": "123...", ...}
  ```

挑战3：响应速度
- 目标：整个流程 < 30秒
- 优化：使用GPT-4o-mini（更快更便宜）而非GPT-4

### 4.3 AI能实现的模块（适合Coding Agent）

✅ 容易实现：
- 文件上传接口
- 对话框UI
- 调用OpenAI API
- PDF字段读取（PyPDF2有明确文档）
- PDF字段写入（PyPDF2有示例代码）

⚠️ 需要调试：
- AI Prompt优化（需要多次测试）
- 各种PDF格式的兼容性
- 错误处理逻辑

## 6. 功能边界（明确不做）

### 6.1 第一版不做
1. 用户系统：不做登录、注册、账户管理
2. 数据存储：不保存用户上传的文件和信息
3. Memory功能：不记住用户的常用信息
4. 复杂编辑：不做PDF编辑器，不支持手动调整格式
5. 模板库：不提供常见表单模板
6. 多人协作：不支持分享、评论等社交功能

### 6.2 为什么不做
- 用户系统和数据存储：增加开发复杂度，且涉及隐私安全
- Memory功能：低频使用场景下，记忆功能价值不大
- 复杂编辑：技术难度高，AI难以实现，与核心价值无关
- 模板库：需要大量内容运营，偏离工具定位

---

## 7. 非功能需求

### 7.1 性能
- 文件上传：< 5秒（10MB以内）
- AI处理：< 30秒
- 文件下载：< 5秒

### 7.2 安全与隐私
- 文件处理完立即删除
- 不存储用户信息
- HTTPS加密传输
- 明确告知用户：不保存任何数据

### 7.3 兼容性
- 浏览器：Chrome 90+, Safari 14+, Edge 90+
- 移动端：iOS Safari, Android Chrome
- PDF格式：符合PDF 1.7标准的可编辑PDF

## 8. 给AI Coding Agent的实施建议

### 8.1 分步实现

Step 1: 基础框架（第1-2天）
```
任务：搭建前后端基础框架
- 前端：创建React项目，实现上传页面UI
- 后端：创建FastAPI项目，实现文件接收接口
- 测试：能否成功上传并接收文件
```

Step 2: PDF处理模块（第3-4天）
```
任务：实现PDF字段读写
- 使用PyPDF2读取PDF的表单字段
- 测试写入数据并生成新PDF
- 处理常见错误（非可编辑PDF等）
```

Step 3: AI集成（第5-7天）
```
任务：接入OpenAI API
- 实现字段识别和信息匹配的Prompt
- 调用API并解析返回结果
- 优化Prompt提升准确率
```

Step 4: 完整流程打通（第8-10天）
```
任务：连接前后端，实现完整用户流程
- 前端发送文件和用户输入到后端
- 后端处理并返回填好的PDF
- 前端触发下载
```

Step 5: UI优化和错误处理（第11-14天）
```
任务：优化用户体验
- 加上loading状态
- 完善错误提示
- 响应式布局调整
- 移动端测试
```

Step 6: 部署和测试（第15-21天）
```
任务：上线和真实场景测试
- 部署前后端
- 用真实PDF文件测试
- 修复发现的bug
- 邀请朋友试用并收集反馈
```
